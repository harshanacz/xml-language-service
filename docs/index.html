TypeScript XML intelligence for any JavaScript or TypeScript app

Add XML-aware features to editors, tools, CLIs, and automation.

Parse XML once, then use nine XML features: completions, hover text, document symbols, folding ranges, formatting edits, rename edits, definitions, references, and XSD diagnostics. The package returns plain TypeScript objects, so you decide how to show or apply the results.

$ npm install xml-language-service
Editor agnostic Use it from an editor, web app, CLI, build step, or tests.
Fault tolerant Broken XML still produces a partial document tree.
XSD aware Xerces-C++ WASM handles schema validation.

Install

Install the package and use an ESM-compatible TypeScript setup.

bash 
npm install xml-language-service
Requires Node.js 18+. The package includes the Xerces WASM assets. No native compiler setup is needed.
tsconfig.json 
{
  "compilerOptions": {
    "module": "NodeNext",
    "moduleResolution": "NodeNext"
  }
}

Quick Start

Create one service instance, parse the XML, then call the features you need.

typescript 
import { getLanguageService } from "xml-language-service";

const ls = getLanguageService();

const xml = `<api name="MultiplyAPI">
  <resource>
    <inSequence>
      <property name="result"/>
    </inSequence>
  </resource>
</api>`;

const doc = ls.parseXMLDocument("file:///api.xml", xml);

const completions = ls.doComplete(doc, { line: 1, character: 3 });
const hover = ls.doHover(doc, { line: 2, character: 6 });
const symbols = ls.findDocumentSymbols(doc);
const [formatEdit] = ls.format(doc, { tabSize: 2, insertSpaces: true });

console.log(symbols[0].name); // "api (MultiplyAPI)"

ls.dispose();

9 Features

Parsing is the foundation: every feature starts from an XMLDocument created by parseXMLDocument(uri, text). On top of that tree, the library offers these nine features.

doComplete(document, position)

Suggests element names, attributes, and matching close tags.

doHover(document, position)

Returns Markdown hover content for elements and comments.

findDocumentSymbols(document)

Returns a nested symbol tree with readable labels and symbol kinds.

getFoldingRanges(document)

Returns collapsible ranges for multi-line elements.

format(document, options)

Returns one full-document text edit with AST-guided indentation.

doRename(document, position, newName)

Renames matching opening and closing tags together.

doDefinition(document, position)

Jumps between matching open and close tag ranges.

findReferences(document, position)

Finds elements with the same tag name across the document.

validate(schemaUri, document)

Runs Xerces-backed XSD validation and returns diagnostics.

How Each Feature Works

The implementation matches the nine-feature list above. The parser is used first, but it is treated as shared infrastructure: it builds the tree, keeps exact source offsets, and lets the feature services return small result objects without mutating the document.

Foundation: parser. parseXMLDocument() uses @xml-tools/parser to build a fault-tolerant XMLDocument. The document stores nodes, attributes, offsets, parent/child links, syntax errors, and helpers such as findNodeAt(offset) and traverse(callback).
Feature 1

Completion

Provides context-aware suggestions while a user or tool is editing XML.

  • Input: a parsed XMLDocument and a zero-based cursor Position.
  • How context is detected: the position is converted to an offset, then the service inspects the text before that offset.
  • After <: it suggests element names. If a schema provider has data, it suggests schema-valid children for the current parent. Without schema data, it scans the document tree and suggests element names already present in the XML.
  • After </: it calls document.findNodeAt(offset) and suggests the current node's matching close tag.
  • Inside an opening tag: it suggests attributes. With schema data, it returns XSD-defined attributes and their types. It also merges attributes already seen on the same tag name. Without schema data, it falls back to xml:lang, xml:space, and xmlns.
  • Output: a CompletionList with items, label, kind, insertText, and optional detail.
Feature 2

Hover

Displays useful information for the XML node under the cursor.

  • Input: a parsed document and cursor position.
  • Node lookup: the position is converted to an offset, then document.findNodeAt(offset) returns the deepest node at that location.
  • Without schema data: element hover shows the tag name, attributes with current values, and child count.
  • With schema data: hover uses the resolved XSD provider to show schema documentation, valid attributes, attribute types, required flags, and valid child elements.
  • Comments: comment nodes can return a comment hover. Root and plain text positions return null.
  • Output: a HoverResult with Markdown contents and a range covering the relevant XML node.
Feature 3

Document Symbols

Builds a hierarchical navigation tree for any client that needs an outline, breadcrumbs, search index, or structure view.

  • Input: a parsed XML document.
  • Traversal: the service recursively walks document.children and then each child element's children.
  • Labeling: symbol labels include identifying attributes in priority order: name, then id, then key. For example, <api name="MultiplyAPI"> becomes api (MultiplyAPI).
  • Hierarchy: child symbols are placed inside the parent symbol's children array, preserving the XML structure.
  • Kind mapping: known XML and WSO2 MI elements are classified as kinds such as class, method, function, property, array, struct, object, and field.
  • Navigation precision: range covers the full element, while selectionRange covers the opening tag name so navigation lands on the tag definition.
Feature 4

Folding Ranges

Identifies XML regions that can be collapsed by an editor, viewer, or custom UI.

  • Input: a parsed document.
  • Traversal: the service walks all element nodes recursively.
  • Line calculation: node start and end offsets are converted to line numbers.
  • Filtering: only multi-line, non-self-closing elements become folding ranges. Self-closing tags and same-line elements are skipped because collapsing them is not useful.
  • Output: a flat FoldingRange[], where each item has startLine, endLine, and kind: "region".
Feature 5

Formatting

Formats XML with AST-guided indentation rather than a fragile line-by-line approach.

  • Input: a parsed document and formatting options: tabSize and insertSpaces.
  • Core approach: the formatter walks parsed element nodes and uses exact source offsets to locate opening tags, closing tags, child regions, and loose content.
  • Indentation: indentation level comes from node depth. The indent unit is either spaces from tabSize or a tab character when insertSpaces is false.
  • One-line XML: nested one-line XML such as <root><child/></root> expands into a readable multi-line structure.
  • Attributes: attributes remain attached to the parent opening tag. Multi-line attributes are normalized onto the opening tag instead of being treated like child content.
  • Mixed content: inline mixed content is preserved when changing whitespace could change meaning, such as <p>Hello <b>world</b>!</p>.
  • Output: one full-document TextEdit. The formatter never mutates the existing AST.
Feature 6

Rename

Renames XML tag pairs atomically by returning the exact text edits a client should apply.

  • Input: a parsed document, cursor position, and newName.
  • Validation: empty names and names containing whitespace or angle brackets are rejected.
  • Target lookup: the cursor position is converted to an offset, then the element under that offset is found.
  • Opening tag edit: the opening tag name range starts after < and spans the current element name.
  • Closing tag edit: for non-self-closing elements, the service searches backward within the node range for </currentName and creates a second edit.
  • Self-closing tags: only one edit is returned because there is no separate closing tag.
  • Output: TextEdit[] or null when rename cannot apply.
Feature 7

Definition

Provides tag-pair navigation between matching opening and closing tags.

  • Input: a parsed document and cursor position.
  • Target lookup: the service finds the element under the cursor.
  • Opening tag behavior: when the cursor is on the opening tag name, the service searches forward for the matching closing tag and returns that range.
  • Closing tag behavior: when the cursor is on the closing side, it returns the opening tag range.
  • Self-closing tags: return null because there is no paired tag to navigate to.
  • Output: a DefinitionResult with document uri and target range, or null.
Feature 8

References

Finds all elements in the same document that share the selected tag name.

  • Input: a parsed document and cursor position.
  • Target lookup: the service finds the element under the cursor and reads its tag name.
  • Traversal: it calls document.traverse(...) and checks every element node.
  • Matching: every element whose name equals the target tag name is returned, including the current element.
  • Output: ReferenceResult[], each with document uri and full element range.
Feature 9

XSD Validation

Validates XML against registered schemas and returns diagnostics that clients can display or process.

  • Input: a registered schema URI and parsed document. The validator reads document.text.
  • Registration: registerSchema(...) compiles a schema and stores it under a URI. If the same URI is registered again, the old validator is disposed and replaced.
  • Engine: validation uses Apache Xerces-C++ compiled to WebAssembly.
  • Why Xerces: Xerces validates with a SAX-style pass, so syntax and schema diagnostics can be reported together instead of requiring a perfect DOM first.
  • Single XSD: pass { uri, xsdText } to registerSchema.
  • Multi-file XSD: pass an imports map whose keys match schemaLocation values used by xs:include or xs:import.
  • Schema-aware support: the same registered or resolved XSD data can enrich completion and hover.
  • Output: Diagnostic[] with range, message, severity, and source. Source is syntax for XML parse errors or xsd for schema violations.

Symbols

Symbols are hierarchical. Labels include identifying attributes such as name, id, and key, which makes large WSO2 MI files easier to scan.

  • <api name="MultiplyAPI"> becomes api (MultiplyAPI).
  • <property name="result"/> becomes property (result).
  • Children stay nested for Outline and Breadcrumb navigation.
symbol result 
[
  {
    name: "api (MultiplyAPI)",
    kind: "class",
    children: [
      {
        name: "resource",
        kind: "method",
        children: [
          { name: "property (result)", kind: "property" }
        ]
      }
    ]
  }
]

Formatter

Formatting is AST-guided. The formatter walks parsed nodes, uses node offsets to rebuild tags, and respects tabSize plus insertSpaces.

  • Nested one-line XML expands into readable indentation.
  • Attributes stay attached to the opening tag.
  • Mixed inline text is preserved instead of being split apart.
format example 
<root><child><leaf/></child></root>
<root>
  <child>
    <leaf/>
  </child>
</root>

XSD Validation

Register schemas once, then validate parsed documents. Built-in schema association can resolve known files, and schema bundles support xs:include and xs:import.

typescript 
await ls.registerSchema({
  uri: "file:///schema.xsd",
  xsdText
});

const diagnostics = await ls.validate("file:///schema.xsd", doc);
Validation is async. The WASM validator is loaded lazily on the first schema validation call.

AST / CST Printer

Two opt-in utilities let you inspect exactly what the parser produced. Neither function affects normal LSP performance — they are only called when you explicitly invoke them.

  • printAST(doc, options?) — prints the XMLNode tree: elements, attributes, and self-closing markers.
  • printCST(doc, options?) — prints the raw Chevrotain grammar rules and tokens captured during parsing.
  • Both accept format: "tree" (default) or format: "json".
  • includePositions: true adds [start..end] character offsets to every node.
  • includeTokens: false hides raw syntax tokens from CST output, showing only rule names.

Available as standalone imports or via the language service object. The JSON format is stable enough for snapshot-based tests.

tree output — printAST 
import { parseXMLDocument, printAST, printCST } from "xml-language-service";

const doc = parseXMLDocument(
  "file:///my.xml",
  `<root><item id="1"/></root>`
);

console.log(printAST(doc));
// Document
//   Element <root>
//     Element <item> /
//       Attribute id="1"

console.log(printAST(doc, { includePositions: true }));
// Document [0..26]
//   Element <root> [0..26]
//     Element <item> / [6..17]
//       Attribute id="1" [12..14]
JSON format — snapshot testing 
const json = JSON.parse(printAST(doc, { format: "json" }));
// {
//   "type": "root",
//   "uri": "file:///my.xml",
//   "children": [{
//     "type": "element",
//     "name": "root",
//     "children": [{
//       "type": "element",
//       "name": "item",
//       "isSelfClosing": true,
//       "attributes": [{ "name": "id", "value": "1" }]
//     }]
//   }]
// }
PrintOptions 
type PrintOptions = {
  format?:           "tree" | "json"; // default "tree"
  indent?:           number;          // default 2
  includePositions?: boolean;         // default false
  includeTokens?:    boolean;         // default true (CST only)
}

Use Cases

This is a general XML service. It does not require a specific editor or protocol. A client can be a UI, a command-line tool, a build process, a test helper, or an extension adapter.

Code Editors

Adapt completions, hovers, symbols, and text edits to editor APIs such as VS Code, Monaco, CodeMirror, or a custom browser editor.

CLIs and CI

Parse and validate XML files in scripts, pre-commit checks, build pipelines, or release gates.

XML Tools

Build formatters, schema-aware inspectors, migration tools, dashboards, or config analysis utilities.

example adapter mapping 
const editorSymbol = {
  label: symbol.name,
  icon: iconFor(symbol.kind),
  range: toEditorRange(symbol.range),
  selectionRange: toEditorRange(symbol.selectionRange),
  children: symbol.children.map(toEditorSymbol),
};

Types

The core shapes are small TypeScript objects that can be mapped into your own UI, protocol, or storage format.

XMLDocument

Root node with uri, text, children, syntax errors, and tree helpers.

XMLNode

Element node data: name, attributes, children, offsets, parent, and self-closing state.

DocumentSymbol

Outline entry with name, kind, range, selectionRange, and recursive children.

Diagnostic

Validation result with range, severity, message, source, and optional code.

Architecture

The data flow is intentionally small and predictable.
