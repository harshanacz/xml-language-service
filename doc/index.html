XML Language Service

A pure-TypeScript library that delivers full XML editing intelligence — completions, hover, symbols, formatting, rename, go-to-definition, find-references, and XSD validation — with no coupling to any specific editor. Drop it into a VS Code extension, a web editor, or a CLI tool.

XSD validation is powered by Apache Xerces-C++ compiled to WebAssembly. Because Xerces uses a SAX streaming parser, it validates as it reads — so you get both syntax errors and schema violations in the same pass, even on broken or incomplete XML.

$ npm install xml-language-service

Context-aware suggestions for elements, attributes, and closing tags. XSD-aware when a schema is loaded.

Rich Markdown hover content showing element name, attributes, and valid children from the XSD.

Full hierarchical outline of the XML document for the editor sidebar tree view.

Re-indents the document and normalizes spacing. Respects tabSize and insertSpaces options.

Rename

Renames both opening and closing tags atomically. Handles self-closing tags correctly.

Syntax + schema errors in one pass via Xerces-C++ WASM. Works even on malformed XML.

Jump between matching open and close tag pairs instantly.

Finds all elements sharing the same tag name across the document.

Multi-file XSD

Full xs:include and xs:import support via SchemaBundle — pass dependent XSD files as a map.

Installation

Install via npm, yarn, or pnpm. The package ships as pure ESM with full TypeScript declarations.

bash
npm install xml-language-service
# or
yarn add xml-language-service
# or
pnpm add xml-language-service

Requires Node.js 18+. The package bundles Xerces-C++ as WASM — no native addons or compiler toolchain needed.

tsconfig requirements

The package is pure ESM. A compatible tsconfig.json:

    "module": "ESNext",
    "moduleResolution": "bundler"
  }

Quick Start

Create one service instance at startup and reuse it. All parsing and editing features are synchronous.

typescript
import { getLanguageService } from "xml-language-service";

const ls = getLanguageService();

const xmlText = `<catalog>
  <book id="bk101">
    <author>Gambardella, Matthew</author>
    <title>XML Developer's Guide</title>
  </book>
</catalog>`;

// 1. Parse — always succeeds, even on broken XML
const doc = ls.parseXMLDocument("file:///catalog.xml", xmlText);

// 2. Completions at cursor position (line 0-based, character 0-based)
const completions = ls.doComplete(doc, { line: 1, character: 3 });
console.log(completions.items.map(c => c.label));
// → ["catalog", "book", "author", "title"]

// 3. Hover
const hover = ls.doHover(doc, { line: 1, character: 3 });
console.log(hover?.contents);
// → "### <book>\n**Attributes:** 1 — `id`\n**Children:** 2"

// 4. Hierarchical symbol outline
const symbols = ls.findDocumentSymbols(doc);

// 5. Format
const [edit] = ls.format(doc, { tabSize: 2, insertSpaces: true });
// edit.newText → properly indented XML

// 6. Rename open + close tag atomically
const edits = ls.doRename(doc, { line: 1, character: 3 }, "item");

// 7. XSD validation (async — loads WASM on first call)
await ls.registerSchema({ uri: "file:///schema.xsd", xsdText: myXsdContent });
const diagnostics = await ls.validate("file:///schema.xsd", doc);

ls.dispose(); // free WASM memory when done

getLanguageService()

The sole public entry point. Returns a stateful service object that owns one internal schema registry and WASM instance. Create one instance per application lifetime.

getLanguageService () → LanguageService sync

Creates and returns a new LanguageService object. No arguments required. The WASM binary is loaded lazily on the first call to validate().

Always call ls.dispose() when you no longer need the service to release the WASM memory held by registered schemas.

parseXMLDocument()

Parses an XML string into an XMLDocument AST. Fault-tolerant — never throws, even on broken or incomplete input.

parseXMLDocument (uri, text) → XMLDocument sync
ParameterTypeDescription
uristringDocument identifier, e.g. "file:///path/to/file.xml". Appears on the returned document and in result objects.
textstringRaw XML source text. Broken or incomplete XML is accepted — a partial tree is built.
Returns

An XMLDocument — a normalized AST with findNodeAt(offset) and traverse(callback) helpers. Pass this object to any other service method.

typescript
// Works on valid XML
const doc = ls.parseXMLDocument("file:///my.xml", "<root><child /></root>");
console.log(doc.children[0].name);   // "root"

// Also works on broken XML — builds a partial tree
const broken = ls.parseXMLDocument("file:///tmp.xml", "<root><unclosed");
console.log(broken.children[0].name); // "root"

doComplete()

Returns context-aware completion items at a given cursor position.

doComplete (document, position, fileName?) → CompletionList sync
ParameterTypeDescription
documentXMLDocumentParsed document from parseXMLDocument().
positionPositionZero-based { line, character } cursor.
fileNamestring?File name used to auto-resolve a built-in schema (e.g. "pom.xml"). Optional.
Completion contexts
  • After </ — suggests the matching close tag. kind: "closeTag"
  • After < or partial tag name — suggests all element names seen in the document (XSD-aware when a schema is loaded). kind: "element"
  • In attribute position — suggests xml:lang, xml:space, xmlns, plus XSD-defined attributes. kind: "attribute"
  • Otherwise — empty list.
typescript
const doc = ls.parseXMLDocument("file:///x.xml", "<root><child/></root>");

const list = ls.doComplete(doc, { line: 0, character: 1 });
list.items.forEach(i => console.log(i.label, "→", i.insertText));
// "root"  → "root>$0</root>"
// "child" → "child>$0</child>"

Returns Markdown hover content for the XML node under the cursor.

fileNamestring?Used to resolve built-in schemas for richer XSD-aware hover content.

findDocumentSymbols()

Builds a hierarchical symbol tree from all element nodes — ideal for an editor outline panel.

Returns

An array of DocumentSymbol. Each symbol has a range (full element), a selectionRange (just the tag name), and nested children.

//   {

//     children: [
//       { name: "book", kind: "element", children: [...] }
//     ]
//   }
// ]

Flat array of FoldingRange. Each has startLine, endLine, and kind: "region".

doRename()

Renames both the opening and closing tags of the element under the cursor atomically.

doRename (document, position, newName) → TextEdit[] | null sync
ParameterTypeDescription
documentXMLDocumentParsed document.
positionPositionCursor on an element tag (open or close).
newNamestringThe replacement tag name.
Returns

Two TextEdits (open + close) for regular elements; one for self-closing. Returns null if newName is invalid or the cursor is not on a named element.

typescript
const edits = ls.doRename(doc, { line: 0, character: 1 }, "container");
// edits[0] → rename open tag   <root>  → <container>
// edits[1] → rename close tag  </root> → </container>

doDefinition()

Navigates between a matching open/close tag pair — "go to definition" for XML tags.

doDefinition (document, position) → DefinitionResult | null sync
Returns

DefinitionResult pointing to the opposite tag. null for self-closing elements or plain text.

typescript
// Cursor on open tag → result points to matching close tag
const def = ls.doDefinition(doc, { line: 0, character: 1 });
// { uri: "file:///x.xml", range: { start: {...}, end: {...} } }

findReferences()

Finds all elements in the document that share the same tag name as the element under the cursor.

findReferences (document, position) → ReferenceResult[] sync
Returns

Array of ReferenceResult — one per matching element, including the cursor element itself. Returns [] if not on a named element.

typescript
const refs = ls.findReferences(doc, { line: 1, character: 3 });
// All <book> elements in the document:
// [{ uri: "file:///catalog.xml", range: {...} }, ...]

XSD Validation — How it works

XSD validation is powered by Apache Xerces-C++ compiled to WebAssembly. Xerces streams through the raw XML text character by character with schema validation enabled. Errors are collected as they are encountered — the parse never needs to finish for errors to be reported.

raw XML text ↓ Xerces SAX stream (parsing + schema validation simultaneously) ↓ syntax errors → source: "syntax" schema errors → source: "xsd"

Behaviour by scenario

Scenariosyntax errorsschema errors
Valid XMLnonenone
Schema violations onlynoneall violations
Syntax error onlyfatal error reportednone — parse stopped
Schema error then syntax errorfatal error reportederrors up to the crash point

Xerces stops at the first fatal syntax error. Everything before that point is still reported — both syntax and schema errors together. This is why you can get XSD violations from a partially malformed file.

Single XSD

Register a schema by URI, then validate any parsed document against it.

typescript
import { getLanguageService } from "xml-language-service";

const ls = getLanguageService();

const xsd = `<?xml version="1.0"?>
<xs:schema xmlns:xs="http://www.w3.org/2001/XMLSchema">
  <xs:element name="root">
    <xs:complexType>
      <xs:sequence>
        <xs:element name="child" type="xs:string"/>
      </xs:sequence>
    </xs:complexType>
  </xs:element>
</xs:schema>`;

// Register schema under a URI key
await ls.registerSchema({ uri: "file:///schema.xsd", xsdText: xsd });

// Validate a document
const doc = ls.parseXMLDocument("file:///my.xml", "<root><child>hello</child></root>");
const errors = await ls.validate("file:///schema.xsd", doc);
console.log(errors); // []

// Schema violation
const bad = ls.parseXMLDocument("file:///bad.xml", "<root><unknown/></root>");
const errs = await ls.validate("file:///schema.xsd", bad);
// [{ message: "...", severity: "error", source: "xsd", range: {...} }]

// Malformed XML — syntax error returned alongside any schema errors found so far
const broken = ls.parseXMLDocument("file:///broken.xml", "<root><unclosed");
const brokenErrs = await ls.validate("file:///schema.xsd", broken);
// [{ source: "syntax", ... }]

ls.dispose();

Multi-file XSD (SchemaBundle)

When your root XSD references other files via xs:include or xs:import, pass a SchemaBundle. The imports keys must exactly match the schemaLocation values used inside the XSD.

typescript — xs:include
// root.xsd contains:  <xs:include schemaLocation="types.xsd"/>

await ls.registerSchema({
  uri:     "file:///root.xsd",
  xsdText: rootXsdContent,
  imports: {
    "types.xsd": typesXsdContent,  // key matches schemaLocation value
  },
});
typescript — xs:import
// company.xsd contains:
//   <xs:import namespace="http://example.com/address"
//              schemaLocation="address.xsd"/>

await ls.registerSchema({
  uri:     "file:///company.xsd",
  xsdText: companyXsdContent,
  imports: {
    "address.xsd": addressXsdContent,  // key matches schemaLocation value
  },
});

Deeply nested imports are supported. Add every dependency (including transitive ones) to the imports map. Xerces resolves them from the map without touching the filesystem.

SchemaBundle type

typescript
interface SchemaBundle {
  entry:    string | Buffer | Blob;                        // root XSD content
  imports?: Record<string, string | Buffer | Blob>;        // dependency files
}

// SchemaInfo (passed to registerSchema)
interface SchemaInfo {
  uri:      string;            // unique key for this schema
  xsdText:  string;            // root XSD content
  imports?: Record<string, string>;  // dependency files (same as SchemaBundle.imports)
}

Built-in Schemas

The service ships with schemas for common file types and auto-associates them by filename or xmlns namespace — no registration needed.

FileNamespace / PatternSchema
pom.xmlhttp://maven.apache.org/POM/4.0.0Maven POM 4.0.0
web.xmlhttp://xmlns.jcp.org/xml/ns/javaeeJava Servlet 3.1

Custom schemas and patterns can be registered with addUserAssociation():

typescript
ls.addUserAssociation({
  pattern: "**/*.config.xml",           // glob pattern
  uri: "file:///my-config-schema.xsd",  // must already be registered
});

// Or match by namespace:
ls.addUserAssociation({
  namespace: "http://example.com/my-ns",
  uri: "file:///my-ns-schema.xsd",
});

Use resolveSchemaForDocument() to check which schema would be auto-resolved for a given file:

typescript
const resolved = ls.resolveSchemaForDocument("pom.xml");
// { uri: "built-in://maven-4.0.0.xsd", xsdText: "...", source: "builtin" } | null

Validation API Reference

registerSchema (info: SchemaInfo) → Promise<void> async

Compiles an XSD and registers it under a URI key. Replaces any existing schema at that URI. Loads the WASM binary on the first call.

FieldTypeDescription
uristringUnique identifier used later in validate() and hasSchema().
xsdTextstringRoot XSD source text.
importsRecord<string, string>?Dependent XSD files for xs:include / xs:import. Keys must match schemaLocation values.
validate (schemaUri, document) → Promise<Diagnostic[]> async

Validates a parsed document against a registered XSD schema in a single Xerces SAX pass. Never throws — always resolves to an array.

ParameterTypeDescription
schemaUristringURI of a previously registered schema.
documentXMLDocumentDocument to validate.
Returns

Empty array means valid. If schemaUri is not registered, returns a single severity: "warning", source: "xsd" diagnostic.

hasSchema (uri: string) → boolean sync

Returns true if a schema is registered under uri.

dispose () → void sync

Disposes all registered validators and releases WASM memory. Call this when the service is no longer needed.

Using XsdValidatorService directly

For low-level control, XsdValidatorService is also exported:

import { XsdValidatorService, SchemaBundle } from "xml-language-service";

// Plain XSD string
const validator = await XsdValidatorService.create(xsdText);

// SchemaBundle (xs:include / xs:import)
const bundle: SchemaBundle = {
  entry:   rootXsdContent,
  imports: { "types.xsd": typesXsdContent },
};
const bundleValidator = await XsdValidatorService.create(bundle);

const diagnostics = await bundleValidator.validate(xmlText); // never throws
bundleValidator.dispose();

LSP Integration

All result types are modeled after the Language Server Protocol. Bridging to an LSP server is straightforward.

✅

Position, Range, DocumentSymbol, FoldingRange, Diagnostic, DefinitionResult, ReferenceResult all mirror LSP types exactly. Only TextEdit uses character offsets instead of Range — convert with an offsetToPosition() helper if needed.

VS Code extension sketch

typescript — extension.ts
import * as vscode from "vscode";
import { getLanguageService } from "xml-language-service";

const ls = getLanguageService();

export function activate(ctx: vscode.ExtensionContext) {
  // Completions
  ctx.subscriptions.push(
    vscode.languages.registerCompletionItemProvider("xml", {
      provideCompletionItems(document, position) {
        const doc  = ls.parseXMLDocument(document.uri.toString(), document.getText());
        const list = ls.doComplete(doc, position);
        return list.items.map(item => {
          const ci = new vscode.CompletionItem(item.label);
          ci.insertText = new vscode.SnippetString(item.insertText);
          return ci;
        });
      }
    }, "<", "/"),

    // Hover
    vscode.languages.registerHoverProvider("xml", {
      provideHover(document, position) {
        const doc   = ls.parseXMLDocument(document.uri.toString(), document.getText());
        const hover = ls.doHover(doc, position);
        return hover ? new vscode.Hover(hover.contents) : undefined;
      }
    }),

    // XSD diagnostics
    vscode.workspace.onDidChangeTextDocument(async e => {
      if (e.document.languageId !== "xml") return;
      const doc        = ls.parseXMLDocument(e.document.uri.toString(), e.document.getText());
      const schemaInfo = ls.resolveSchemaForDocument(e.document.fileName);
      if (!schemaInfo) return;

      if (!ls.hasSchema(schemaInfo.uri)) {
        await ls.registerSchema({ uri: schemaInfo.uri, xsdText: schemaInfo.xsdText });
      }
      const diags = await ls.validate(schemaInfo.uri, doc);
      diagnosticCollection.set(e.document.uri, diags.map(d => new vscode.Diagnostic(
        new vscode.Range(d.range.start.line, d.range.start.character,
                         d.range.end.line,   d.range.end.character),
        d.message,
        d.severity === "error" ? vscode.DiagnosticSeverity.Error : vscode.DiagnosticSeverity.Warning
      )));
    })
  );
}

export function deactivate() { ls.dispose(); }

Architecture

A high-level view of how data flows through the library.

TextDocument (uri + raw string) │ ▼ parseXMLDocument() → @xml-tools/parser (Chevrotain CST) │ ▼ XMLDocumentImpl (normalized node tree) │ ├─── doComplete() completion items ├─── doHover() markdown hover ├─── findDocumentSymbols() hierarchical outline ├─── getFoldingRanges() collapsible regions ├─── format() indented text edits ├─── doRename() tag pair text edits ├─── doDefinition() matching tag range ├─── findReferences() same-name elements │ └─── SchemaProvider (URI → XsdValidatorService) │ └─── XsdValidatorService → Xerces-C++ WASM (SAX parse + XSD validation, single pass)

Key design decisions

  • Fault-tolerant parsing — the underlying parser (@xml-tools/parser) builds a partial tree on broken input and never throws. All service methods work on incomplete documents.
  • Xerces SAX over libxml2 DOM — libxml2 requires a full DOM parse before XSD validation can run, making it useless on malformed XML. Xerces's SAX streaming approach validates as it reads, so syntax errors and schema violations are reported in the same pass even when the document is incomplete.
  • Immutable documents — services read XMLDocument but never mutate it. format() operates on the raw text string only.
  • LSP-compatible shapes — Position, Range, DocumentSymbol, FoldingRange, Diagnostic all mirror LSP types, making adaptation to an LSP server straightforward.
  • SchemaBundle via emscripten::val — the Xerces C++ bridge uses a polymorphic emscripten::val parameter so a single validate() binding accepts both a plain XSD string and a { entry, imports } bundle object without any branching in the binding layer.
